# ADR-0001: Expor API do Operator para gerenciar Collections

## Status
Accepted

## Context
Hoje os clientes criam collections diretamente via API do Qdrant.
Isso gera:
- configurações inconsistentes
- ausência de backups automáticos
- risco operacional em restores
- ausência de auditoria

## Decision
O operator irá expor uma API HTTP interna para:
- criar collections a partir de templates
- listar collections
- listar backups
- restaurar collections

Essa API **não executa ações diretamente**, apenas cria/atualiza CRDs
(`QdrantCollection`, `QdrantCollectionBackup`, `QdrantCollectionRestore`).

## Consequences
### Positivas
- Padronização de collections
- Backups habilitados por padrão
- Restore controlado e auditável
- Menor acoplamento do cliente ao Qdrant

### Negativas
- Operator assume responsabilidade adicional
- Nova superfície de API para manter
- Requer autenticação interna

## Alternatives Considered
1. Clientes chamarem Qdrant diretamente
   - Rejeitado: sem governança e backup por padrão

2. Criar scripts externos
   - Rejeitado: sem reconciliação declarativa

3. Helm templates para collections
   - Rejeitado: Helm não reconcilia runtime state
